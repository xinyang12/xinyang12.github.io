发行一个自己的ERC-20代币

发表于 2020-05-05 更新于 2020-05-12

ERC-20 代币是什么

要弄清楚这个问题，首先要明白代币是什么。以太坊上的代币其实就是一种加密货币，可以用作消费，交易，股权证明等等。在技术本质上，其实就是一个简单的智能合约。ERC-20 是以太坊上实现代币的一种规范协议，是目前发行量最大的代币协议。ERC 代表 Ethereum Request For Comments，20则代表 ID 号码。类似的代币协议还有 ERC 223， ERC 777 等等。前几年大火的 ICO（Initial Coin Offering，首次代币发售）就是用户使用 ETH 购买项目方发行的以太坊代币，从而持有其股权，有的代币甚至涨了几百几千倍。其中大部分都是实现 ERC-20 规范的

既然 ERC-20 是一种规范协议，那就说明实现它的合约，都遵守了某种协议。就像 Android 系统一样，Google 制定系统的规范，而各家厂商都会按照规范发行自己的定制版系统。虽然各家都有不同，但是一个符合开发规范的 App 是可以在各个系统上运行的。

ERC-20 规范

ERC-20 标准包含了 6 条必须遵守的强制规则和 3 条可选规则。

强制规则包括：

  • totalSupply
  • balanceOf
  • transfer
  • transferFrom
  • approve
  • allowance
  • Transfer（事件，Event）
  • Approval（事件，Event）

可选规则包括：

  • Name
  • Symbol
  • Decimal

下面来一一解读这些规则

可选规则

Name

function name() public view returns (string memory)

Name 即是代币的名称，类似于 BitCoin，Ethereum 这种。

Symbol

function symbol() public view returns (string memory)

Symbol 是代币的简称，像上面的 BitCoin 这样的名称太长，我们就可以用 BTC 来指代这种代币。BTC、ETH 这种就是代币的 Symbol。

Decimal

function decimals() public view returns (uint8)

代币单位小数点后的位数，类似于人民币，最小单位是分，基础单位是元，那么人民币中的 Decimal 就是 2。在以太坊中，1 Ether = 10 ^ 18 wei，那么 Decimal 就是 18。该字段的默认值是 18。

强制规则

totalSupply

function totalSupply() public view returns (uint256)

该代币的总供应量，要注意这里配置的总量要考虑前面的 decimal 的因素，如果说想要总共 1000 个 代币，而 decimal 配置为 2，那么这里 totalSupply 就应该是 1000 * (10 ^ 2) = 100,000。

balanceOf

function balanceOf(address _owner) public view returns (uint256 balance)

传入 _owner 的地址，返回其所拥有的该代币的余额

transfer

function transfer(address _to, uint256 _value) public returns (bool success)

这个方法由付款方调用，向 _to 地址转入 _value 数量的代币。该方法中必须触发 Transfer 事件。

transferFrom

function transferFrom(address _from, address _to, uint256 _value) public returns (bool success)

approve

function approve(address _spender, uint256 _value) public returns (bool success)

allowance

function allowance(address _owner, address _spender) public view returns (uint256 remaining)

这三项放在一起来说，因为它们之前有很大的关联。在现实生活中，我们会有一些一段时间固定要做的事情，比如说每个月都要还信用卡账单，有时候比较忙我们可能就会忘记还款了，那么银行就提供一项代扣款的功能，每到还钱的时候，银行会自动扣除钱款。

同样的，这三个函数所做的就是这些功能。首先，调用者调用了 approve 方法，传入 _spender 的地址和允许代扣款的数额 _value，这样 _spender 就有权利从调用者的账户转走最多 _value 数额的钱款。其次，在 _spender 被授权之后，它就可以调用 transferFrom 方法，_from 参数就是前面的被代理人，_to 是接收人，_value 则是钱款。最后，allowance 方法返回 _spender 可以转移 _owner 钱财的最大数量，也就是 approve 方法中设置的 _value。

这一套机能内部是用 mapping 结构实现的：

mapping (address => mapping (address => uint256)) public allowed;

其中，外部 mapping 的 key address 是被代理人的地址（类比于上面设置自动扣款的客户），内部 mapping 的 key address 是代理人的地址（类比于上面有权利扣款的银行）。

Transfer （事件，Event）

代币转账时必须要触发 Transfer 事件。

Approval （事件，Event）

调用上面的 approve 函数成功时必须要触发 Approval 事件。

再来测试一下 transfer 方法，复制第二个账户地址，填入 transfer 的框框中，后面再加上逗号和要转的代币数量，比如，框中输入：0x8472C29BF4b2386fBb08518F9CA92468b6bE9263, 10000

注意这里前面是第二个账户的地址，中间是英文的逗号，后面是要转的数量，我们这里转 10000 个，然后一定记得将上面的账户一栏再换成第一个账户地址，因为只有第一个账户地址才有代币～

输入完成后，点击 transfer，如果没有报错的话就说明转账成功。然后我们再次通过 balanceOf 函数查询他们的余额，发现其余额分别变成了 90000 和 10000，转账成功～

其他的方法，大家可以自己试试，一定要注意不要把调用者搞错了～

真实测试网络部署

上面是在虚拟网络中部署的，我们简单部署测试一下，发现没有什么问题。不过我们只在这里部署就太没意思了。我们现在准备开始在 Rinkeby（其他的测试网络也可以） 测试网络中部署，不要看它叫测试网络，其实它和以太坊主网的机制是完全一样的，也是有矿工在进行实时挖矿操作。我们在测试网络上部署就类似于在主网部署，也是全世界人民都能看到的～

首先我们需要在环境一栏选择 Injected Web3（这里的前提是安装了 MetaMask插件，并且切换到 Rinkeby 网络，同时有测试 ETH。这里就先不详细说了，可以在网络上搜搜，教程很多，或者以后也可能出一篇教程），然后选择我们的 PolarCoin 合约，点击 Deploy 会跳出 MetaMask 的确认窗口，点击确认，即可发送部署交易，稍等片刻（等待交易被包含在区块）即可看到合约已经部署成功。
这样我们就已经在 Rinkeby 测试网络中成功部署了我们的合约。

到这里是不是发现和前面的虚拟网络也没啥太大区别，别着急，我们现在打开 MetaMask 插件

在自定义代币这里将我们的之前的代币合约地址复制到地址栏里，稍等一下，下面就会自动出现代币符号（symbol）和精确小数点（decimals），这是 MetaMask 自动在 Rinkeby 网络中搜索到的。

点击下一步，看到我们的代币余额是 1000.00，注意这个值是 totalSupply / decimals 之后的值

点击添加代币，这样我们就能在 MetaMask 中管理我们自己发行的代币了，是不是有一点小小的成就感～

主网部署

主网部署其实和测试网络部署的步骤一模一样，只不过有一点很重要，主网部署是要花费真金白银的（真实的 ETH ）～ 如果你有 ETH 并且想在主网部署自己的代币，当然是可以的～

总结

花了几天时间断断续续写完了这篇文章，简单介绍了从 0 开始创建自己的 ERC-20 代币的过程。大家有一点要注意，那就是 ERC-20 是一个规范，它只规定了大体的框架，具体的实现还是要参照具体的场景。

ERC-20 是比较基础的代币规范，这几年还有 ERC-777， ERC-223 等规范，都是在其基础上进行完善和补充，以后有机会可以介绍一下其他的合约规范。
